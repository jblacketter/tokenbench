# Token-Efficiency Pattern Library

Generalizable patterns for spending fewer tokens without losing capability. Every
pattern here is **paired with a reproducible measurement** — the project's first
principle is *measure, don't guess*, so no pattern ships as untested advice.

## How to read a pattern

Each pattern doc follows the same template:

- **Family** — Data Delivery, Query, or Agent Loop.
- **Problem** — the situation that burns tokens.
- **Forces** — what makes it tricky (why the naive fix isn't free).
- **Solution** — the pattern.
- **When to use / When not** — boundaries, so it isn't applied blindly.
- **Token impact (measured)** — a table **generated by the harness**, not typed by
  hand. The numbers come from `tokenbench/patterns/scenarios.py` and are regenerated
  by `python -m tokenbench patterns --markdown`. A test asserts the docs are in sync
  with the harness, so prose can never silently drift from the numbers.
- **Tradeoffs** — latency, ergonomics, quality.
- **Related patterns.**

Measurements are either **measured** (real captured production data, e.g. the
MCP-vs-CLI case study) or **estimated** (the ~4-chars/token proxy applied to real
fixture payloads, offline and deterministic). Each table says which.

## Reproduce the numbers

```bash
python -m tokenbench patterns              # results table
python -m tokenbench patterns --markdown   # doc-ready markdown
python -m tokenbench patterns --family Query
```

## Patterns

| Family | Pattern | Anchor |
|--------|---------|--------|
| Data Delivery | [Filtered Response](data-delivery/filtered-response.md) | shape the payload to the question |

| Agent Loop | [Snapshot Budget](agent-loop/snapshot-budget.md) | bound the snapshots carried across a loop |

More patterns named in the roadmap (Inline All, Progressive Disclosure, Targeted
Eval, Two-Phase Execution, Context Pruning, Parallel Fan-Out) extend these three
families and can be added with the same measured template.
